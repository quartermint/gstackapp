## Problem Statement

gstack is a 25-skill CLI toolkit that turns Claude Code into a virtual engineering team. It works. Garry ships 10K-20K lines/day with it. But it's local-only, single-user, and invisible -- no dashboard, no team memory, no cross-repo intelligence. The skills die when the terminal closes.

The question: what does gstack look like as a platform product? Not "gstack in a browser" but a genuinely new thing -- a cognitive quality pipeline that runs as a service, remembers everything, learns across repos, and makes the 51% quality paradox (more AI = more bugs) disappear.

## What Makes This Cool

Nobody has built a cognitive CI/CD platform. CodeRabbit does line-by-line review. Qodo does dependency analysis. Anthropic's tool catches AI bugs. All single-lens tools.

gstack's insight is that code quality is a pipeline of different thinking modes:
1. **CEO review** -- challenges the premise ("is this the right problem?")
2. **Eng review** -- locks architecture, tests, failure modes
3. **Design review** -- catches AI slop, spacing issues, hierarchy problems
4. **QA** -- opens a real browser, clicks real buttons
5. **Security audit** -- OWASP Top 10 + STRIDE threat modeling

That's not five flavors of "review the diff." That's five fundamentally different cognitive functions applied in sequence. The platform version makes this visible, persistent, and compounding.

The 10x vision adds:
- **Cross-repo intelligence** -- "Your auth pattern in repo A has a bug we caught in repo B"
- **AI team memory** -- "Last time you made this architecture choice, it caused X"
- **Multiplayer skills** -- one person's CEO review feeds another's eng review
- **Quality Arena** -- skill pipelines compete via Elo scoring, best emerge per context

## Recommended Approach

**Approach B first, evolve toward C.**

Build Mission Control 3.0 as the full platform with these layers:

### Layer 1: GitHub Integration (the entry point)
- GitHub App installs in one click
- **Required permissions:** `contents: read`, `pull_requests: write`, `checks: write` on connected repos
- **Webhook security:** All incoming webhooks verified against `X-Hub-Signature-256` before processing
- On every PR: run cognitive pipeline (CEO, Eng, Design, QA, Security)
- **Pipeline failure handling:** Stages fail independently. On stage failure (API timeout, rate limit, model error), that stage is marked SKIPPED with error reason. Partial results are posted immediately. Pipeline never blocks on a single stage failure.
- **Comment behavior:** Bot posts one comment per PR, updates it in-place on force-push or re-run (no comment spam)
- Post a single "Cognitive Check" comment with:
  - Per-stage verdicts (PASS / FLAG / BLOCK / SKIPPED)
  - Overall confidence score (0-100)
  - Merge risk delta (how much riskier is this PR vs the repo's rolling 30-day average quality score)
  - "Seen in your other repos" cross-repo insights (scoped to same org)

### Layer 2: Dashboard (the "whoa" moment)
- **Pipeline view:** Visualize each PR flowing through cognitive stages (like GitLab CI/CD but the stages are thinking modes)
- **Quality trends:** Line charts showing quality scores over time, per repo and cross-repo
- **Review history:** Every cognitive review ever run, searchable, filterable
- **Team insights:** Who's shipping what, quality patterns per contributor -- *deferred to post-launch iteration* (team retro view is not required for prototype)

### Layer 3: Cross-Repo Intelligence (the moat)
- Every review is stored with embeddings (pgvector), scoped to the user's org (single-org for v1; cross-org federation deferred to Approach C)
- **Data retention:** Embeddings store review summaries and finding descriptions, not raw source code. Code snippets in findings are limited to the diff context (not full files). Orgs can delete their data via settings.
- "Seen in your other repos" pulls precedents from your org's connected repos
- Pattern detection: "This auth pattern was flagged in 3 of your repos"
- Decision memory: "Last time you chose X over Y, the outcome was Z"

### Layer 4: Skill Marketplace (the ecosystem) -- POST-PROTOTYPE
*Deferred to post-launch iteration. Not required for tweetable MVP.*
- Browse and install community skills (plug into SkillsMP/ClawHub ecosystem)
- Create custom cognitive stages (your team's domain-specific reviews)
- Rate and rank skills per repo (the seed of Approach C's Quality Arena; cross-org context deferred to Approach C)

### Tech Stack

| Component | Technology | Why |
|-----------|-----------|-----|
| Frontend | Next.js 14 (App Router) | Fast, SSR, familiar to gstack community |
| Backend | Next.js API routes + tRPC | Type-safe frontend API; webhook receiver is thin (enqueue only) |
| Worker | Fly.io long-running process | Pipeline stages run async via BullMQ jobs (5+ AI calls, 30-120s total) |
| Database | Postgres + pgvector (Neon or Supabase) | Relational + embeddings for cross-repo memory; single-org scope for v1 |
| Queue | BullMQ + Upstash Redis | Serverless Redis, works with Vercel and Fly.io |
| AI Runtime | Claude Sonnet 4.6 (CEO/Eng/Design/QA) + Opus 4.6 (Security only) | Cost control: Sonnet for volume stages, Opus for high-stakes security audit |
| Auth | GitHub OAuth | Users are already on GitHub |
| Hosting | Vercel (frontend + webhook receiver) + Fly.io (worker) | Split: Vercel for fast frontend deploys, Fly.io for long-running pipeline jobs |
| GitHub Integration | GitHub App (webhooks) | PR events trigger pipeline runs |

**Architecture note:** Webhook receiver on Vercel accepts the event, verifies `X-Hub-Signature-256`, enqueues a BullMQ job, and returns 200 immediately. The Fly.io worker picks up the job, runs 4 cognitive stages (QA deferred), and posts the GitHub comment on completion. This avoids Vercel's function timeout limits.

**Worker retry policy:** 3 attempts per job, exponential backoff (5s, 30s, 120s). Failed jobs go to a dead letter queue. Dashboard surfaces stuck/failed jobs in a "Pipeline Health" indicator. If all 4 stages fail after retries, bot posts a "Pipeline Error" comment with diagnostic info.

**Auth flow:** GitHub OAuth callback -> upsert User record + link to GitHub Org -> set encrypted session cookie -> redirect to dashboard. Users table added to data model (below). Org-level app installation grants access to all installed repos for cross-repo search.

**Model versions:** Platform uses fixed model versions (Sonnet 4.6 / Opus 4.6) regardless of local gstack configuration. Custom model selection is post-prototype.

**Quality score formula:** Weighted average of per-stage verdicts: PASS=1.0, FLAG=0.5, BLOCK=0.0, SKIPPED=excluded. Overall score = (sum of stage scores / count of non-skipped stages) * 100. Merge risk delta = current PR score minus repo's rolling 30-day average score (negative = riskier than usual).

**Cognitive Check comment template:**
```markdown
## Cognitive Check | Mission Control

| Stage | Verdict | Summary |
|-------|---------|---------|
| CEO Review | PASS | Problem is well-scoped, addresses real user need |
| Eng Review | FLAG | Missing error handling in auth middleware (line 47) |
| Design Review | PASS | UI changes follow existing patterns |
| QA | COMING SOON | -- |
| Security | PASS | No OWASP violations detected |

**Quality Score:** 75/100 | **Merge Risk:** +8 (slightly riskier than your 30-day average)

**Seen in your other repos:**
- Similar auth middleware pattern in `api-service` was flagged for missing token expiry check (PR #142, 2 weeks ago)

---
*Powered by [Mission Control](https://mission-control.dev) | [Dashboard](https://app.mission-control.dev/org/repo/pr/123)*
```

### Data Model (Core)

```
Users (GitHub OAuth identity, session management)
  -> OrgMemberships (many-to-many with Organizations)

Organization (GitHub org, billing, budget_remaining)
  -> Repositories (GitHub repos connected)
    -> PullRequests (tracked PRs)
      -> PipelineRuns (one per PR event)
        -> StageResults (CEO, Eng, Design, Security; QA deferred)
          -> Findings (individual issues found)
          -> Embeddings (pgvector for cross-repo search)
  -> Skills (installed cognitive stages) -- post-prototype
  -> QualityTrends (aggregated metrics over time, computed by background job)
  -> TeamMembers (GitHub users linked to org)
```

## Cost Model v1

Running 5 AI stages per PR on every repo is the main cost driver. Default configuration:
- **CEO, Eng, Design, QA stages:** Claude Sonnet 4.6 (~$3/1M input tokens, fast)
- **Security stage:** Claude Opus 4.6 (~$15/1M input tokens, thorough)
- **Estimated cost per PR:** ~$0.15-0.40 depending on diff size
- **Free tier:** 50 PRs/month per org (covers most solo builders and small teams)
- **Pro tier:** Unlimited PRs, Opus for all stages, $29/month per org
- **Token budget enforcement:** At the BullMQ queue layer. Jobs exceeding budget are queued until next billing cycle.
