    **Auth routes** (packages/api/src/routes/auth.ts):

    `POST /magic-link`: Zod validate `{ email: string }`. Check `isKnownUser(email)` -- return 403 if not. Generate token, store hash in `magicLinkTokens` table (nanoid for id), send email. Return 200 `{ sent: true }`.

    `GET /verify`: Query params `token` and `email`. Look up unexpired, unused token for that email in `magicLinkTokens`. Verify with `verifyMagicLinkToken`. If valid: mark token `usedAt = new Date()`, upsert user in `users` table (create if first login, update lastLoginAt), create `userSessions` row (nanoid as id = cookie value, 7-day expiry per D-02), set httpOnly secure cookie `gstack_session` via `setCookie()` from `hono/cookie` (httpOnly: true, secure: NODE_ENV==='production', sameSite: 'Lax', maxAge: 7 * 24 * 60 * 60, path: '/'). Redirect to `/`.

    `GET /me`: This route is mounted INSIDE the auth-protected area (under `/api/auth/me`, protected by authMiddleware). Read the user from `c.get('user')` (set by authMiddleware). Return `{ user: { id, email, role } }`. If no user on context (should not happen since middleware guards), return 401. This endpoint is consumed by App.tsx to determine whether to show LoginPage, OperatorHome, or admin dashboard.

    **Auth middleware** (packages/api/src/auth/middleware.ts):

    Use `createMiddleware` from `hono/factory`. Three-path auth per D-01, with Tailscale-User-Login header fallback (per resolved open question on Funnel IP masking):

    Path 0 -- Tailscale Funnel header: Check `c.req.header('tailscale-user-login')`. If present, this is a tailnet user accessing via Funnel (Funnel sets this header for authenticated tailnet users). Call `resolveRole(headerValue)`. If role found, upsert user in DB, set `c.set('user', { id, email: headerValue, role, source: 'tailscale' })`, call `next()`.

    Path 1 -- Tailscale IP: Get remote IP from `c.req.header('x-forwarded-for')` or socket address. Call `whoisByAddr(ip)`. If returned, call `resolveRole(tsUser.loginName)`. If role found, upsert user in DB, set `c.set('user', { id, email, role, source: 'tailscale' })`, call `next()`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet -> Tailscale Funnel -> Hono API | Untrusted HTTP from any origin hits auth middleware |
| Tailscale tailnet -> Hono API | Semi-trusted: user is on tailnet but identity needs verification via whois |
| Browser cookie -> Session lookup | Cookie value is an opaque token; must validate against DB |
| Email inbox -> Magic link token | Token delivered via email; attacker could intercept |
| Tailscale-User-Login header -> Role resolution | Header set by Funnel; trusted only because Funnel is the reverse proxy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | Spoofing | authMiddleware | mitigate | Tailscale whois over Unix socket (authoritative identity); cookie verified against userSessions table with expiry check; Tailscale-User-Login header trusted only from Funnel (header spoofing mitigated by Funnel being the sole reverse proxy) |
| T-17-02 | Spoofing | magic-link.ts | mitigate | HMAC-SHA256 tokens (32 bytes entropy), 15-min expiry, single-use (delete after verify), timingSafeEqual comparison |
| T-17-03 | Spoofing | magic-link.ts | mitigate | Rate limit magic link requests (max 3 per email per 15 min via DB count check) |
| T-17-04 | Tampering | auth routes | mitigate | httpOnly + secure + sameSite=Lax cookie prevents XSS/CSRF theft; session ID is opaque nanoid |
| T-17-05 | Repudiation | auth middleware | mitigate | All auth events (login, magic link request, token verify) logged via pino with userId, email, source, timestamp |
| T-17-06 | Information Disclosure | auth middleware (D-04) | mitigate | getUserScope enforces WHERE user_id=? on all operator queries; admin bypass explicit |
| T-17-07 | Denial of Service | magic-link.ts | accept | SendGrid rate limits (100 free/day) naturally cap email flood; known-user check (isKnownUser) blocks unknown emails |
| T-17-08 | Elevation of Privilege | roles.ts | mitigate | Hardcoded allowlist in env vars (D-03); no self-registration; unknown emails get 403 |
</threat_model>
