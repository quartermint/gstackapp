This is the Garry Tan gstack philosophy applied to code review. The CEO stage challenges the PREMISE of the change, not just the implementation. Sections to include:

1. **Role definition**: "You are the CEO reviewer -- a strategic product thinker who challenges the premise of every change. You don't review code quality (that's the Eng stage). You review whether this change SHOULD exist."
2. **Core questions**: Does this change make sense for users? Is this the right abstraction? Is the complexity justified by the value? Is there a simpler way to achieve the same outcome? Does this change align with the project's direction?
3. **What to look for**: New abstractions (are they premature?), dependency additions (is the dependency justified?), architecture changes (is the migration path clear?), feature additions (is this the right feature to build next?), configuration complexity (is this adding operational burden?).
4. **How to use tools**: Start by reading the PR diff summary. Use `list_files` to understand project structure. Use `read_file` to examine new files, architecture docs, README. Use `search_code` to find similar patterns already in the codebase.
5. **Category values**: "premise", "abstraction", "direction", "complexity", "dependency", "scope"

**Step 3: Create `packages/api/src/__tests__/prompts.test.ts`.**

This test verifies all 5 prompt files exist and meet formatting/size requirements. It does NOT need the test-db setup since it only reads files. Use ESM-compatible `import.meta.url` for directory resolution:

```typescript
import { readFileSync } from 'node:fs'
import { resolve, dirname } from 'node:path'
import { fileURLToPath } from 'node:url'
import { describe, it, expect } from 'vitest'

const currentFile = fileURLToPath(import.meta.url)
const currentDir = dirname(currentFile)

const STAGES = ['ceo', 'eng', 'design', 'qa', 'security'] as const
const PROMPTS_DIR = resolve(currentDir, '../pipeline/prompts')

describe('Stage prompt files', () => {
  for (const stage of STAGES) {
    describe(stage, () => {
      const promptPath = resolve(PROMPTS_DIR, `${stage}.md`)

      it('exists and is readable', () => {
        const content = readFileSync(promptPath, 'utf-8')
        expect(content.length).toBeGreaterThan(0)
      })

      it('contains JSON format instructions with verdict and findings', () => {
        const content = readFileSync(promptPath, 'utf-8')
        expect(content).toContain('"verdict"')
        expect(content).toContain('"findings"')
        expect(content).toContain('"severity"')
      })

      it('contains verdict definitions (PASS, FLAG, BLOCK)', () => {
        const content = readFileSync(promptPath, 'utf-8')
        expect(content).toContain('PASS')
        expect(content).toContain('FLAG')
        expect(content).toContain('BLOCK')
      })

      it('is substantial enough for prompt caching', () => {
        const content = readFileSync(promptPath, 'utf-8')
        const wordCount = content.split(/\s+/).length
        // Opus stages (ceo, security) need 4096+ tokens (~1500+ words)
        // Sonnet stages (eng, design, qa) need 2048+ tokens (~1000+ words)
        const minWords = (stage === 'ceo' || stage === 'security') ? 1500 : 1000
        expect(wordCount).toBeGreaterThanOrEqual(minWords)
      })
    })
  }
})
```

Note: This test file imports only from `node:fs`, `node:path`, `node:url`, and `vitest`. It does NOT import any application modules, so it will not trigger any module-level side effects. The vitest setupFiles (test-db.ts) will still run because vitest.config.ts loads it globally -- but that is benign since this test makes no DB calls. If there are issues with the global setup loading config (it sets env vars first), it should be safe because test-db.ts sets all required env vars before any config module loads.
