    **`packages/api/src/db/schema.ts`:**
    Define Drizzle tables per the research Pattern 2.

    `captures` table: `id` (text PK, nanoid), `rawContent` (text not null), `type` (text enum ['text','voice','link','image'] default 'text'), `status` (text enum ['raw','pending_enrichment','enriched','archived'] default 'raw'), `projectId` (text nullable), `userId` (text nullable -- PLAT-02), `createdAt` (integer timestamp not null), `updatedAt` (integer timestamp not null).

    `projects` table: `slug` (text PK), `name` (text not null), `tagline` (text nullable), `path` (text not null), `host` (text enum ['local','mac-mini'] default 'local'), `lastScannedAt` (integer timestamp nullable), `createdAt` (integer timestamp not null), `updatedAt` (integer timestamp not null).

    Add indexes: `captures` index on `projectId`, index on `status`, index on `userId`. `projects` index on `host`.

    Do NOT define FTS5 tables in the Drizzle schema (this would cause push conflicts -- see research Pitfall 3).

    **`packages/api/drizzle.config.ts`:**
    ```typescript
    import { defineConfig } from 'drizzle-kit';
    export default defineConfig({
      schema: './src/db/schema.ts',
      out: './drizzle',
      dialect: 'sqlite',
      dbCredentials: { url: process.env.DB_PATH ?? './data/mission-control.db' },
    });
    ```

    **Generate migrations:** Use `drizzle-kit generate` to create `0000_initial_schema.sql` from the schema. Then create a custom migration for FTS5.

    **`packages/api/drizzle/0001_fts5_search.sql`:** (custom migration)
    Create FTS5 virtual table `captures_fts` per research Pattern 2. Content table references `captures`. Column: `raw_content`. Create insert/update/delete triggers to keep FTS in sync. Include a project_metadata_fts for project names and taglines for search across projects.

    Create `drizzle/meta/_journal.json` that tracks both migrations in order.

    **`packages/api/src/db/index.ts`:**
    Per research Code Examples (Database Initialization). Create better-sqlite3 Database instance. Set pragmas: `journal_mode = WAL`, `busy_timeout = 5000`, `synchronous = NORMAL`, `cache_size = -64000`, `foreign_keys = ON`. Create Drizzle instance with schema. Run migrations on startup from `drizzle/` folder. Export `db` and `sqlite` (raw instance for FTS5 queries). Use `DB_PATH` env var with default `data/mission-control.db`. Ensure data directory exists with `mkdirSync`.

    **`packages/api/src/db/queries/captures.ts`:**
    Query functions using Drizzle ORM:
    - `createCapture(data: CreateCapture)`: Generate nanoid, insert with timestamps, return full capture.
    - `getCapture(id: string)`: Select by id, throw AppError NOT_FOUND if missing.
    - `listCaptures(query: ListCapturesQuery)`: Select with optional where clauses for projectId, status, userId. Apply limit/offset. Return array.
    - `updateCapture(id: string, data: UpdateCapture)`: Update fields, set updatedAt. Return updated capture. Throw NOT_FOUND if missing.
    - `deleteCapture(id: string)`: Delete by id. Throw NOT_FOUND if missing.

    **`packages/api/src/db/queries/search.ts`:**
    Per research Code Examples (FTS5 Search Query).
    - `searchCaptures(query: string, limit?: number)`: Execute raw SQL against `captures_fts` with BM25 ranking. Join with `captures` table for full data. Return array of `{ id, rawContent, type, projectId, rank, createdAt }`.
    - Sanitize FTS5 query input: strip special FTS5 operators (`AND`, `OR`, `NOT`, `NEAR`, quotes) to prevent injection. Wrap terms in double quotes for phrase matching if needed.

    All query functions accept the raw Drizzle `db` instance as a parameter OR use the singleton -- pick whichever is cleaner for testing (parameter injection is preferred for testability).
