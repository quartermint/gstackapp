  <action>
    Create packages/harness/src/skills/runner.ts implementing the skill runner per D-11, D-12, D-13.

    Define SkillRunInput interface:
    ```typescript
    interface SkillRunInput {
      manifest: SkillManifest
      provider: LLMProvider
      adapter: ToolAdapter
      model: string
      executeTool: (name: string, input: Record<string, unknown>) => Promise<string>
      maxIterations?: number  // default 10
      maxTokens?: number  // default 4096
      signal?: AbortSignal
      userMessage?: string  // optional initial user message
    }
    ```

    Define SkillResult interface:
    ```typescript
    interface SkillResult {
      output: unknown  // validated against manifest.outputSchema
      tokenUsage: number
      durationMs: number
    }
    ```

    Export async function runSkill(input: SkillRunInput): Promise<SkillResult>

    Implementation steps:
    1. Validate tool support: for each tool in manifest.tools, call adapter.mapToolName(tool). If any throws, re-throw with skill name context (D-13 fail-fast).
    2. Build tool definitions: create ToolDefinition[] for each manifest.tools canonical tool, then map through adapter.mapToolSchema.
    3. Resolve prompt: if manifest.prompt starts with './' or '/' or ends with '.md', readFileSync the path. Otherwise use as inline text.
    4. Run tool_use loop (replicate stage-runner.ts lines 202-262 pattern):
       - Initialize messages array with user message
       - Loop up to maxIterations
       - Call provider.createCompletion with mapped tools
       - On end_turn: extract text, parse output, break
       - On tool_use: extract tool blocks, for each block reverse-map name via adapter, call executeTool, map result via adapter.mapToolResult, push assistant + user messages in CRITICAL order (assistant first, then user with tool results)
    5. Parse output text as JSON, return SkillResult with output, tokenUsage, durationMs.

    Note: The executeTool function is passed in by the caller (not built into the runner). This keeps the runner decoupled from any specific tool implementation. The caller wires up the actual tool execution (sandbox tools, shell tools, etc).

    Write tests FIRST using mock LLMProvider (vi.fn() returning controlled CompletionResult sequences). Test the full loop with 1-2 tool iterations then end_turn.
  </action>
