       Import `tool` from "ai" and `z` from "zod". Import query functions from db/queries/.

       Export `createChatTools(dbInstance: DatabaseInstance, userId: string)` that returns an object of AI SDK tool definitions:

       ```typescript
       import { tool } from "ai";
       import { z } from "zod";
       import type { DatabaseInstance } from "../db/index.js";
       import { listProjects, getProject } from "../db/queries/projects.js";
       import { listCaptures, createCapture } from "../db/queries/captures.js";
       import { searchUnified } from "../db/queries/search.js";
       import { getActiveFindings } from "../db/queries/health.js";
       import { sessions } from "../db/schema.js";
       import { desc } from "drizzle-orm";
       ```

       Tools to define (all per RESEARCH Pattern 1 -- RAG via tool calling):

       a. `listProjects` -- List all tracked projects with status and health
          - parameters: z.object({})
          - execute: calls listProjects(db, {}), maps to summary objects

       b. `getProjectStatus` -- Get detailed status for a specific project
          - parameters: z.object({ slug: z.string() })
          - execute: calls getProject + getActiveFindings, returns project + findings + health summary
          - On not found: return { error: "Project not found" } (don't throw)

       c. `getRecentCaptures` -- Get recent captures, optionally by project
          - parameters: z.object({ projectSlug: z.string().optional(), limit: z.number().default(10) })
          - execute: calls listCaptures with filters

       d. `getImessageExtracts` -- Get iMessage conversation extracts (per D-06)
          - parameters: z.object({ limit: z.number().default(20) })
          - execute: query captures table WHERE sourceType = "imessage", ORDER BY createdAt DESC

       e. `searchMC` -- Search across all MC content
          - parameters: z.object({ query: z.string() })
          - execute: calls searchUnified(sqlite, query, { limit: 10 })

       f. `createCapture` -- Create a capture on Ryan's behalf (per D-05)
          - parameters: z.object({ content: z.string(), projectSlug: z.string().optional() })
          - execute: calls createCapture(db, { rawContent: content, projectId: projectSlug, userId, sourceType: "bella" })
          - Returns the created capture

       g. `getRecentSessions` -- Get recent Claude Code sessions
          - parameters: z.object({ limit: z.number().default(5) })
          - execute: query sessions table ORDER BY startedAt DESC LIMIT

       Each tool wraps execute in try/catch, returns { error: message } on failure.

       Define BELLA_SYSTEM_PROMPT as a const string (per RESEARCH Pattern 4):
       ```
       You are Mission Control, Ryan's personal operating environment.
       You are talking to {user.displayName}, a {user.role === "owner" ? "the owner" : "team member"}.

       RULES:
       - ALWAYS use tools to look up project status, captures, and activity. Never guess or make up project details.
       - When asked "what's Ryan working on?", use listProjects and getRecentSessions to see current activity.
       - When asked about a specific project, use getProjectStatus to get real data.
       - When asked to remember or capture something, use createCapture.
       - Present information conversationally, not as raw data dumps. Summarize and contextualize.
       - If you don't have data to answer a question, say so honestly. Do not hallucinate project names or statuses.
       - You can see iMessage conversation extracts -- use getImessageExtracts when asked about recent discussions.
       - After gathering data with tools, synthesize a helpful response. Do not call the same tool twice with identical parameters.

       CONTEXT:
       - Ryan works in serial sprints -- one project gets intense focus for days/weeks, then he moves on.
       - Mission Control tracks: projects (git repos), captures (thoughts/notes), sessions (Claude Code), health checks, knowledge (CLAUDE.md files).
       - The MC API serves: dashboard (web), CLI, MCP server, iOS companion, and this chat.
       - When explaining how MC works, be clear and educational -- {user.displayName} is learning the platform.
       ```

       Use template literal to inject user.displayName and role at request time.

       Export `createChatRoutes(getInstance: () => DatabaseInstance, getConfig: () => MCConfig | null)`:

       POST /chat handler:
       a. Parse `{ messages }` from request body. Validate messages is a non-empty array. Return 400 if invalid.
       b. Check `getLmStudioStatus()`. If health !== "ready", return 503 JSON: `{ error: { code: "LLM_UNAVAILABLE", message: "MC intelligence is warming up" } }`.
       c. Resolve user from request headers via `resolveUser(c.req.raw.headers, config?.users ?? [])`.
       d. Build system prompt with user context.
       e. Create LM Studio provider: `createOpenAI({ baseURL: "http://100.123.8.125:1234/v1", apiKey: "not-needed" })`.
       f. Call `streamText({ model, system, messages, tools: createChatTools(getInstance(), user.id), maxSteps: 5 })`.
       g. Return `result.toDataStreamResponse()` (AI SDK v6 uses toDataStreamResponse, not toUIMessageStreamResponse).

    2. Wire route in `packages/api/src/app.ts`:

       - Add route BEFORE the intelligence routes (to avoid slug matching conflicts, same pattern as Phase 37 decision):
         `.route("/api", createChatRoutes(getInstance, () => config ?? null))`
