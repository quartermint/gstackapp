Refactor `packages/web/src/components/operator/OperatorHome.tsx`:

Major refactor to become the chat-thread orchestrator. State machine driven by SSE events:

```typescript
type OperatorViewState =
  | { phase: 'idle' }
  | { phase: 'clarifying'; requestId: string; questions: Array<{ question: string; answer?: string }> }
  | { phase: 'briefing'; requestId: string; brief: ExecutionBriefData; questions: Array<{ question: string; answer: string }> }
  | { phase: 'running'; requestId: string; currentStage: string; startedAt: string }
  | { phase: 'complete'; requestId: string; report: VerificationReport }
  | { phase: 'error'; requestId: string; errorType: string; message: string; report?: VerificationReport }
  | { phase: 'escalated'; requestId: string }
```

Orchestration logic:
1. Start in `idle` phase — show IntakeForm at top.
2. When IntakeForm submits successfully, the SSE event `operator:clarification:question` arrives:
   - Transition to `{ phase: 'clarifying', requestId, questions: [{ question }] }`.
   - Render IntakeForm (now read-only/disabled), then ClarificationThread below it.
3. When operator answers via ClarificationThread, POST to `/api/operator/${requestId}/clarify-answer`.
   - Next SSE event either adds another question or `operator:brief:generated` arrives.
4. When `operator:brief:generated` arrives:
   - Transition to `{ phase: 'briefing', requestId, brief, questions }`.
   - Render chat thread (intake + clarification) above, then ExecutionBrief card.
5. When operator clicks "Approve & Start" — POST to `/api/operator/${requestId}/approve-brief`.
   - `operator:brief:approved` arrives, then `operator:progress` events start.
   - Transition to `{ phase: 'running', requestId, currentStage, startedAt: new Date().toISOString() }`.
   - Render chat thread above, OperatorProgressBar below, decision gates inline.
6. When operator clicks "Request Changes" — POST to `/api/operator/${requestId}/reject-brief`.
   - Back to clarifying state.
7. When `operator:complete` arrives:
   - Transition to `{ phase: 'complete', requestId, report }` if `operator:verification:report` also arrived.
   - Render VerificationReport card.
8. When `operator:error` arrives:
   - Transition to `{ phase: 'error', requestId, errorType, message }`.
   - Render ErrorCard. Handle actions:
     - 'wait': POST retry-timeout, back to running.
     - 'escalate': POST escalate, transition to `{ phase: 'escalated' }`.
     - 'request-changes': Go back to clarifying with new question.
     - 'retry': POST retry, back to running.
9. When `operator:gate` arrives during running phase: render gate card inline (reuse existing GateCard pattern from PipelineProgress.tsx).

SSE connection: Open EventSource to `/api/sse` when a requestId is active. Process events same as existing PipelineProgress pattern but with the extended event types.
