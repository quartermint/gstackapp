## The Thesis

MC today is a dashboard. v2.0 makes it a brain.

The Mac Mini isn't just hosting an API — it's running an intelligence daemon that learns, compounds knowledge, and makes you smarter every interaction. Foundation Models on iOS extends that intelligence to the edge. Claude Code sessions are the hands. The dashboard is the face.

### Phase 32: Hybrid Search Intelligence

**Inspired by:** tobi/qmd
**Addresses deferred:** "Semantic/vector search via embeddings"

MC's current search is basic FTS5 with optional Gemini query rewriting. qmd proves you can build production-grade hybrid search with the exact same stack (better-sqlite3, SQLite FTS5, sqlite-vec). This phase builds a proper search pipeline.

**What qmd teaches us:**
- sqlite-vec two-step query pattern (vector query first, JOIN second — sqlite-vec hangs otherwise)
- Content-addressable storage (SHA-256 hash dedup, renamed files don't need re-embedding)
- RRF fusion with position-aware blending (trust retrieval for top results, reranker for deep results)
- Context tree: annotate projects with descriptions returned alongside search results
- Query expansion via local LLM (typed: `lex:` → BM25, `vec:` → vector, `hyde:` → hypothetical doc)

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| SRCH-01 | sqlite-vec extension with local embedding model via LM Studio | qmd uses embeddinggemma-300M; MC should use LM Studio's embedding endpoint |
| SRCH-02 | Content-addressable storage for all indexed content | SHA-256 hash dedup like qmd — captures, commits, knowledge all keyed by content hash |
| SRCH-03 | Hybrid BM25 + vector search with Reciprocal Rank Fusion | `score = sum(weight / (k + rank + 1))`, k=60, original query gets 2x weight |
| SRCH-04 | Query expansion via LM Studio replacing Gemini query rewriting | Local LLM generates typed expansions; falls back to direct FTS5 if LM Studio unavailable |
| SRCH-05 | Context annotations on projects returned with search results | Like qmd's context tree — "this project is a Swift iOS app for flight recording" |
| SRCH-06 | CLAUDE.md content indexed in unified search (not separate LIKE query) | Knowledge search currently uses substring LIKE — must be FTS5 + vector indexed |
| SRCH-07 | Cross-encoder reranking via LM Studio for top candidates | Optional quality pass — rerank top 20-40 candidates for precision |

**Success Criteria:**
1. Semantic query "how does the capture pipeline work" returns relevant results even without keyword matches
2. Search results include fused BM25 + vector scores
3. LM Studio generates query expansions locally (no Gemini dependency for search)
4. CLAUDE.md content searchable through unified search
5. Content-addressable storage prevents duplicate embeddings

**Key decision:** Do we use LM Studio's OpenAI-compatible embedding endpoint or bundle a dedicated embedding model (like qmd's embeddinggemma-300M via node-llama-cpp)? LM Studio is simpler but requires the model to be loaded; dedicated model is independent but adds a dependency.

**From project-nomad:** Embedding at scale (35+ project knowledge bases, 800+ Capacities imports, ongoing captures) needs a proper async job queue, not `queueMicrotask`. Consider BullMQ + Redis (NOMAD's approach) or a lighter SQLite-based job queue. NOMAD proves this works at Wikipedia-scale with SSE progress broadcasting.

### Phase 33: Capture Intelligence Engine

**Inspired by:** google/langextract concepts
**Evolves:** existing Gemini categorization pipeline

Today's capture enrichment is a single Gemini API call that classifies a capture to one project with a confidence score. langextract shows that extraction + grounding + multi-pass can be dramatically more useful — and the concepts port to any stack.

**What langextract teaches us:**
- Few-shot extraction defined by examples, not training — provide 10 example captures with correct assignments
- Post-hoc grounding via difflib alignment — show which words triggered categorization without LLM producing offsets
- Multi-pass for recall — run extraction twice and merge non-overlapping results
- Prompt validation — check that few-shot examples actually work before using them in production

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| CAP-01 | Few-shot categorization with user-correctable examples | User corrections to AI categorization stored as new few-shot examples |
| CAP-02 | Multi-pass extraction: project refs, action items, ideas, links | Not just "which project" but "what kind of capture is this and what should happen next" |
| CAP-03 | Post-hoc grounding — highlight which words triggered each extraction | Deterministic alignment (like langextract), not LLM-generated offsets |
| CAP-04 | Confidence calibration from user feedback (track correction rate per project) | If user corrects "openefb" → "sfr" often, adjust threshold for those projects |
| CAP-05 | Local LLM fallback for categorization when Gemini unavailable | LM Studio as backup enrichment engine (works offline from cloud) |
| CAP-06 | Prompt validation at startup — verify few-shot examples still produce correct extractions | Like langextract's pre-flight alignment check |
| CAP-07 | Extraction types: project_ref, action_item, idea, link, question | Beyond just project assignment — classify the capture's intent |
| CAP-08 | Capacities import bridge — read backup ZIPs, extract tweets/weblinks/notes, enrich | ~/Capacities_backup/ has automated daily ZIPs with 800+ bookmarks. Import, fetch tweet content (X API/scrape), run through enrichment. Bridge until MC replaces Capacities entirely. |
| CAP-09 | iMessage passive monitoring — poll chat.db for conversations with specified contacts | Filter by contact (Bella), extract action items/ideas/project refs. "Go to where the user is." Requires Full Disk Access or helper app. |
| CAP-10 | Tweet content fetching — resolve bare URLs to full tweet text + thread context | 644 Capacities tweets are bare URLs. Must fetch actual content for enrichment to work. X API or web scraping. |
| CAP-11 | Batch-save UX — handle rapid-fire captures without friction | Jan 6 mega-batch: 42 tweets saved in 1 minute. Pipeline must handle bursts gracefully. |

**Success Criteria:**
1. User corrections to AI categorization improve future predictions (tracked, not discarded)
2. Dashboard shows grounded highlights on captures ("these words matched project X")
3. Action items and ideas auto-extracted and surfaced in project cards
4. Enrichment works offline using LM Studio when Gemini is unreachable
5. Few-shot examples validated at server startup
6. Capacities import enriches 800+ existing bookmarks with context, intent, and project assignment
7. iMessage conversations with Bella surface as captures before Ryan opens MC
8. Tweet URLs resolve to full text content, not bare links

### Phase 34: Knowledge Compounding

**Stolen from:** EveryInc/compound-engineering-plugin
**MC's learning system**

Every.to's compound engineering thesis: each unit of work should make subsequent work easier. Their `/ce:compound` + `learnings-researcher` loop is the mechanism. MC already has session hooks, knowledge aggregation, and MCP tools — the infrastructure exists to make every Claude Code session leave the system smarter.

**What CE teaches us:**
- Solutions registry with structured YAML frontmatter (problem type, root cause, tags, severity)
- Auto-capture from session outcomes (MC already has session stop hooks)
- Learnings-researcher: before any new planning, search past solutions for relevant precedent
- Cross-session knowledge: what was learned fixing a bug in project A helps project B

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| COMP-01 | Solutions registry — `solutions` table with structured metadata | YAML-style: module, problem_type, symptoms, root_cause, tags, severity |
| COMP-02 | Auto-capture from Claude Code session stop hooks | Extract decisions, fixes, and patterns from significant sessions |
| COMP-03 | Learnings surface in MCP session startup banner | Search solutions DB for project-relevant precedent at session start |
| COMP-04 | Cross-session knowledge — solutions from any project searchable by all sessions | MCP `cross_project_search` extended to include solutions |
| COMP-05 | Compound score — track which solutions get referenced and by which sessions | Reuse rate metric on dashboard |
| COMP-06 | Solutions indexed in hybrid search (Phase 32 dependency) | Solutions searchable alongside captures, commits, knowledge |

**Success Criteria:**
1. Session stop hooks auto-generate solution docs from significant sessions (>5 commits or >30min)
2. Session startup MCP banner includes relevant past learnings
3. Dashboard shows compound score (knowledge reuse rate over time)
4. Solutions searchable through unified search and MCP tools
5. Solutions have structured metadata enabling filtering by problem type, project, severity

**Key decision:** How much automation vs. human curation? CE's `/ce:compound` is a manual trigger. MC could auto-generate solution candidates from session hooks and queue them for quick human review (accept/edit/dismiss) — matching the "persist first, review later" pattern.

### Phase 35: Active Intelligence Daemon

**The Mac Mini becomes a brain**
**Addresses deferred:** "AI-generated narrative summaries", "Smart routing with learning"
**Inspired by:** Foundation Models design patterns (constrained generation, tool calling)

LM Studio has been sitting on the Mac Mini doing health probes. It's time to activate it. Foundation Models shows the design patterns: constrained generation for guaranteed structure, tool calling for actions. Apply these patterns to LM Studio.

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| DAEMON-01 | Local LLM query expansion replacing Gemini query rewriting | LM Studio generates typed search expansions (lex/vec/hyde) |
| DAEMON-02 | "Previously on..." AI-generated narrative for project context restoration | Local LLM summarizes recent commits + captures + session outcomes per project |
| DAEMON-03 | Smart session routing with learning from historical outcomes | Track session tier vs. outcome; suggest optimal routing |
| DAEMON-04 | Scheduled intelligence — daily digest at 6am, weekly pattern analysis | Cron-style generation, results cached in DB, dashboard pulls on load |
| DAEMON-05 | Local LLM tool calling — MC defines tools the model can invoke | Tools: search, project lookup, capture enrichment, knowledge query |
| DAEMON-06 | Constrained generation via JSON schema for all local LLM outputs | Like Foundation Models' @Generable — guaranteed structure, no free-form parsing |
| DAEMON-07 | Intelligence cache with TTL — generated content cached and served without re-generation | Narratives cached for 1h, digests for 12h, routing suggestions for session duration |
| DAEMON-08 | Adaptive context injection based on model size | Small models get compact summaries, large models get full context. Stolen from NOMAD's RAG_CONTEXT_LIMITS. |

**Success Criteria:**
1. Opening a project card shows AI-generated "Previously on..." narrative (local LLM)
2. Session startup banner includes routing suggestion based on historical patterns
3. Daily digest generated at 6am with overnight activity summary
4. All local LLM outputs use JSON schema constrained generation
5. Intelligence generation never blocks API responses (cached + async)

**Architectural note:** LM Studio exposes an OpenAI-compatible API. MC's intelligence daemon should abstract over the specific model — today it's Qwen3-Coder-30B, tomorrow it might be a smaller dedicated model for each task (embedding model + chat model + reranker, like qmd does). The daemon manages model lifecycle: warm models for active tasks, cool unused ones.

### Phase 36: iOS Edge Intelligence

**Inspired by:** Apple Foundation Models framework (iOS 26)
**Depends on:** Phase 29-30 (iOS app must exist)

Foundation Models gives iOS apps a free, on-device, offline ~3B parameter model with constrained decoding and tool calling. MC's iOS companion goes from "capture and sync" to "capture, understand, and sync intelligently."

**What Foundation Models enables:**
- Constrained decoding via `@Generable` — guaranteed structured output (project slug, confidence, extraction type)
- On-device classification in <2 seconds, no network needed
- Tool calling for local device actions (calendar context, contacts, location)
- ~4096 token context window (sufficient for single-capture processing)

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| EDGE-01 | On-device structured capture via Foundation Models constrained decoding | @Generable-style capture classification (project, confidence, type) |
| EDGE-02 | Pre-sync enrichment — classify + extract on device before syncing | Simple captures enriched locally, reducing Mac Mini load |
| EDGE-03 | Smart routing — simple captures stay on-device, complex route to Mac Mini | High-confidence single-project → local; ambiguous/multi-project → server |
| EDGE-04 | Offline intelligence — all capture processing works without network | Foundation Models + local queue = full offline functionality |
| EDGE-05 | Context enrichment from device sensors | City-level location, time-of-day, source app, connectivity state |
| EDGE-06 | Context window management — summarization strategy for long captures | Like FoundationChat's approach: summarize when approaching 4096 token limit |

**Success Criteria:**
1. Captures classified on-device within 2 seconds using Foundation Models
2. High-confidence captures (>0.8) enriched entirely on-device, never hit Mac Mini for classification
3. Complex captures sync to Mac Mini with on-device pre-classification as hint
4. All capture processing works offline
5. Capture metadata includes device context (location, time, source app)

**Deployment note:** Foundation Models requires iOS 26 + Apple Intelligence enabled. Phase 29-30 target iOS 17+. Edge intelligence is an enhancement layer — captures still work without Foundation Models, they just skip on-device enrichment.

### Phase 37: Proactive Intelligence

**MC stops being pull-only**

Today's morning experience: open MC, scroll, piece together what happened. v2.0 morning experience: open MC, read the AI-generated digest, know exactly what matters and what to do first.

**Requirements:**

| ID | Requirement | Notes |
|----|-------------|-------|
| PROACT-01 | Morning digest — AI-generated "what happened since you last looked" | Local LLM generates from overnight commits, captures, findings, session outcomes |
| PROACT-02 | Stale capture triage — surface uncategorized/unacted captures for review | Captures older than 7d without project assignment → triage prompt |
| PROACT-03 | Activity pattern detection | "You captured 5 openefb ideas this week but haven't committed in 12 days" |
| PROACT-04 | Session pattern insights | "Your most productive sessions start after 10am and last 45min" |
| PROACT-05 | Cross-project insight surfacing | "cocobanana and openefb both reference MapLibre — shared pattern?" |
| PROACT-06 | Insight dismissal — dismiss/snooze insights that aren't actionable | Prevents insight fatigue; dismissed insights don't resurface |

**Success Criteria:**
1. Dashboard morning view shows AI-generated digest (generated overnight by DAEMON-04)
2. Stale captures surface proactively with suggested actions
3. Activity patterns visible without manual analysis
4. Cross-project patterns detected and surfaced
5. All proactive intelligence generated locally (no external API dependency)

## Key Architectural Decisions (Pending)

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Embedding model | LM Studio endpoint vs. dedicated model (node-llama-cpp) | LM Studio endpoint | Simpler; MC doesn't need to manage GPU allocation. If LM Studio unavailable, skip vector search (graceful degradation). |
| Solutions storage | SQLite table vs. markdown files (like CE) | SQLite table | MC is API-first. Files work for CE because it's a Claude Code plugin reading local files. MC needs to serve solutions via API to dashboard, MCP, and iOS. |
| Intelligence generation | On-demand vs. scheduled | Scheduled + cached | Never block API responses with LLM inference. Generate async, cache with TTL, serve from cache. Like qmd's `llm_cache` table. |
| Foundation Models minimum iOS | iOS 26 only vs. iOS 17+ with optional FM | iOS 17+ with optional FM | Don't gate iOS app on iOS 26 adoption. FM is an enhancement layer. |
| Few-shot example storage | Config file vs. database | Database (examples table) | Examples evolve from user corrections. DB enables API-driven management. |
| Query expansion model | Same model as chat (Qwen3-Coder-30B) vs. dedicated small model | Start with Qwen3-Coder-30B, migrate to dedicated | qmd fine-tuned a 1.7B model for expansion. MC starts with what's available, optimizes later. |
| iMessage access | Direct chat.db polling vs. helper app vs. Shortcuts vs. BlueBubbles | Direct chat.db on Mac Mini (if iCloud synced) | Simplest if Messages syncs to Mini. Needs Full Disk Access. Falls back to MacBook helper if not. |
| Capacities import | One-time import vs. ongoing bridge | Ongoing bridge with deprecation target | Import from daily backup ZIPs until MC capture pipeline is reliable enough to replace Capacities. |
| Tweet content fetching | X API (paid) vs. web scraping vs. Crawl4AI | Crawl4AI (already on Mac Mini) | Crawl4AI Docker container is running. Use it to resolve tweet URLs to content. No new API costs. |
| Async job queue | queueMicrotask vs. BullMQ+Redis vs. SQLite job table | SQLite job table (start), BullMQ if needed | Avoid new dependencies until scale demands it. SQLite can queue embedding jobs. NOMAD validates BullMQ at Wikipedia-scale if we outgrow SQLite. |
| Relationship context | Flat contacts vs. social graph | Start flat (specified contacts in config) | iMessage monitoring starts with configured contacts (Bella). Social graph understanding is future. |

## The Compound Effect

Each phase makes the next more powerful:

1. **Phase 32** (search) gives MC the ability to find anything across all sources
2. **Phase 33** (capture intelligence) gives MC the ability to understand what you capture
3. **Phase 34** (knowledge compounding) gives MC the ability to learn from experience
4. **Phase 35** (intelligence daemon) gives MC the ability to think proactively
5. **Phase 36** (edge intelligence) gives MC the ability to understand at the point of capture
6. **Phase 37** (proactive intelligence) gives MC the ability to tell you what matters

The end state: every time you open Mission Control, you're smarter than you were 3 seconds ago — not because you read data, but because MC already understood it for you.
